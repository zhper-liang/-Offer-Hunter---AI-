# 个人求职辅助系统 - 简历项目描述

## 项目概述

基于 RAG + Agent 架构的全链路 AI 求职辅助平台，覆盖知识库沉淀、智能简历生成、JD 匹配分析、面试准备、语音模拟面试五大场景。系统采用 LangGraph StateGraph 实现有状态 Agent 编排，支持 10+ LLM 提供商运行时切换，通过 RAG 检索增强确保生成内容基于真实经历。

**规模**：后端 Python 68 模块，前端 React/TypeScript 38 模块，15+ Agent 工具，6 套简历模板，支持 4 种文档格式解析。

## 技术难点与解决方案

### 难点一：复杂任务的 Agent 状态管理

**问题**：简历生成需要多轮工具调用（检索知识库 → 生成各段落 → 组装结构化 JSON），传统 ReAct 循环在任务中断、状态回溯、并发工具调用时存在状态丢失问题。

**解决方案**：
- 采用 LangGraph StateGraph 替代自定义 ReAct 循环，定义强类型 AgentState（messages、current_iteration、max_iterations、is_finished、final_answer）
- 设计条件边路由逻辑：Agent Node 判断是否有 tool_calls，有则路由到 Tool Node 执行，无则检查迭代次数和完成标志决定是否终止
- 集成 MemorySaver 检查点机制，支持会话级状态持久化和恢复
- 实现 Thread-based 会话隔离，多用户并发场景下状态互不干扰

**技术权衡**：选择 LangGraph 而非自建 ReAct 的原因是状态图天然支持分支、循环、中断恢复，且与 LangChain 生态深度集成，工具调用格式自动适配。

### 难点二：结构化输出与多模板渲染

**问题**：LLM 输出不可控，直接生成 HTML/PDF 容易格式错误；简历需要支持多种风格模板且数据无损切换。

**解决方案**：
- 设计 Pydantic 强类型简历数据模型 ResumeData，包含 PersonalInfo、WorkExperience、Education、SkillGroup、Project、Certification、CustomSection 等嵌套结构
- LLM 只负责填充结构化 JSON 字段，通过 format_resume 工具输出完整 ResumeData
- 前端 6 套 React 模板（Professional/Tech/Academic/Creative/Executive/Minimalist）消费同一数据源
- 导出时后端根据同一 JSON 渲染 PDF（fpdf2）/ DOCX（python-docx）/ Markdown

**技术权衡**：选择 JSON Schema 而非 Markdown 的原因是结构化数据支持字段级编辑、模板切换、条件渲染，而 Markdown 只能整体替换。

### 难点三：多 LLM 提供商统一接入

**问题**：不同厂商 API 协议差异大（OpenAI 兼容 / Claude 原生 / 自定义），切换成本高，流式和工具调用支持程度不一。

**解决方案**：
- 设计双层 Provider 抽象：LangChain Provider 层使用 ChatOpenAI / ChatAnthropic 统一封装，Custom Provider 层实现 LLMProvider 基类的 chat_with_tools() 接口
- 消息格式转换层：内部消息格式 ↔ LangChain 消息格式（HumanMessage/AIMessage/ToolMessage/SystemMessage）双向转换
- 工具绑定适配：通过 bind_tools() 将工具列表绑定到模型，自动处理不同厂商的 tool_calls 格式差异
- 流式降级机制：优先使用 astream() 流式推送，失败时自动降级到 ainvoke() 阻塞模式

**技术权衡**：选择 LangChain ChatModels 而非直接调用各厂商 SDK 的原因是统一封装降低维护成本，且 LangChain 社区持续跟进各厂商 API 变更。

### 难点四：RAG 检索质量优化

**问题**：简单固定长度分块导致上下文断裂，检索召回率低；扫描件 PDF 无法提取文字。

**解决方案**：
- 三级层次分块策略（Heading → Paragraph → Sentence）：先按 Markdown 标题分割，超长段落按双换行分割段落，再超长按中英文标点分割句子
- 小块合并机制：低于 100 token 的块与相邻块合并，避免碎片化
- 上下文重叠：相邻块添加 100 token 重叠，保持语义连续性
- 查询意图路由器 QueryRouter：分析用户 query 类型（resume/project/comparison/summarization），自动选择 doc_type 过滤和复杂度评估
- 上下文压缩器 ContextCompressor：按 relevance_score 排序，控制总 token 在 6000 以内，避免 LLM 上下文窗口溢出
- PDF OCR 支持：检测扫描件（有效字符占比 < 5%），自动触发 PyMuPDF + pytesseract OCR 提取文字

**技术权衡**：选择 ChromaDB 而非 Milvus/Qdrant 的原因是本地部署零依赖，适合个人使用场景，且 LangChain 原生集成。

### 难点五：实时流式交互体验

**问题**：Agent 多轮执行耗时长（检索 + 生成 + 格式化），用户等待焦虑；需要实时展示工具调用状态。

**解决方案**：
- 后端 SSE（Server-Sent Events）推送 7 种事件类型：text 文本回复、thinking 思考过程、tool_start 工具调用开始、tool_result 工具执行结果、tool_error 工具执行错误、resume_data 结构化简历数据、iteration 迭代轮次
- 前端 EventSource API 实时渲染，thinking 事件展示 Agent 思考过程，tool_start/tool_result 展示工具调用链路
- 流式工具调用：LangChainProvider 的 _chat_with_tools_streaming 方法实现文本 token 边收边 yield，工具调用就绪后一次性 yield
- 流式降级：astream 失败时自动降级到 ainvoke 阻塞模式，保证功能可用性

**技术权衡**：选择 SSE 而非 WebSocket 的原因是简历生成是单向推送场景，SSE 更轻量且浏览器原生支持 EventSource API。

### 难点六：语音面试实时性

**问题**：语音交互需要低延迟双向通信，STT/TTS 厂商 API 延迟各异。

**解决方案**：
- WebSocket 全双工通信，支持音频流实时上传和文本/音频流实时下发
- VoicePipeline 管线设计：audio_to_text（讯飞/百度 STT）→ Agent 处理 → text_to_audio_stream（流式 TTS）
- 双厂商支持：讯飞 WebSocket API 实时语音识别 + 百度语音 API，根据配置自动切换
- 流式 TTS 分块推送，降低首字延迟

### 难点七：文档格式兼容性

**问题**：用户上传的简历/项目文档格式多样（PDF/DOCX/TXT/Markdown），编码和结构差异大。

**解决方案**：
- 统一文档解析器 DocumentLoader，根据扩展名自动选择解析器
- PDF 解析：PyPDF2 提取文字层，扫描件自动触发 OCR（PyMuPDF + pytesseract）
- DOCX 解析：python-docx 提取段落，保留标题层级信息（Heading 1-6 → Markdown #）
- TXT 解析：chardet 自动检测编码（UTF-8/GBK/GB2312 等），避免乱码
- OCR 检测算法：有效字符占比（中文 1 字符 = 1.0 权重，英文 1 字符 = 0.5 权重）< 5% 或绝对字数 < 50 则判定为扫描件

## 架构设计亮点

### Agent 工具系统双格式兼容

同时支持 LangChain @tool 装饰器新格式和 BaseTool 抽象类旧格式，通过 LangChainToolAdapter 自动适配。工具按职责分为 5 类 15+ 个：
- 知识库工具：search_knowledge_base（查询路由 + 检索 + 压缩）、list_documents、delete_document
- 简历工具：generate_section、format_resume、export_resume、update_module_order
- JD 工具：search_jd（多维度匹配）、save_resume_to_jd、list_all_jd
- 面试工具：generate_questions、evaluate_answer、provide_feedback
- 模块工具：update_module_order

### RAG 检索管线设计

完整检索管线：QueryRouter 意图分析 → Embedding 编码 → ChromaDB 向量搜索 → 相关性分数过滤（阈值 0.3）→ 去重（doc_id + chunk_index）→ ContextCompressor 上下文压缩 → assemble_context 组装带来源标注的上下文。

### 两层配置系统

环境变量 .env 存储 API Key 等敏感信息（启动时加载），运行时配置 settings.json 支持前端热修改无需重启（通过 _reload_settings() 热加载）。配置变更实时生效，无需重启服务。

### 简历模块拖拽排序

前端 Zustand + Immer 管理模块顺序状态，拖拽调整后同步写入 resumeData.module_order，Agent 端 update_module_order 工具可持久化保存用户偏好到 settings.json。

## 量化成果

- 支持 10+ LLM 提供商运行时切换（DeepSeek/智谱/Moonshot/通义千问/零一万物/硅基流动/Claude/OpenAI/Ollama/Custom）
- 6 套简历模板共享同一数据源，模板切换零数据丢失
- 3 种导出格式（PDF/DOCX/Markdown）
- 5 类 15+ Agent 工具
- 7 种 SSE 事件类型实时推送
- 4 级层次分块策略（Heading → Paragraph → Sentence → Merge）
- 支持 4 种文档格式（PDF/DOCX/TXT/Markdown）
- PDF 扫描件 OCR 自动识别（PyMuPDF + pytesseract）
- 中英文混合 token 估算（中文 0.75 token/字，英文 0.25 token/字）
- 检索相关性分数阈值过滤（默认 0.3）
- 上下文压缩控制 token 消耗（默认 6000 token）
